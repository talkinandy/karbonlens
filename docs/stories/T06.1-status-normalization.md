## 1. User story

As Andy, I want `projects.status` and `registries.status` to hold canonical
enum values (`active` / `pipeline` / `suspended` / `flagged`) instead of raw
Verra strings, so that T09 transparency scoring distinguishes operational
projects from in-review ones and T11–T12 filter/badge UI has a stable,
well-known vocabulary.

## 2. Context & rationale

The T06 code audit (and later T11/T12 spec audits) flagged that T06's
partial `STATUS_MAP` fell through to `"suspended"` for any unmapped Verra
string. Live Verra query (2026-04-21) returned 11 distinct `resourceStatus`
values for Indonesia, of which only 4 were mapped — and of those fall-throughs,
the majority were in-review pipeline states (e.g. `Verification approval
requested` for Katingan, `Late to verify` for Rimba Raya). Net effect:

- 29 / 64 Indonesian projects were labelled `suspended` when ≤2 actually were.
- T09 `transparency_score` filters on `registries.status = 'active'`, and the
  scraper was writing raw Verra strings (`Registered`, `Under development`, …)
  to `registries.status`, so zero rows matched and every project hit the
  "registry_count≥1 but no active" bucket → flat transparency=55 across all 64.
- T11 explorer filter chip would have exposed 11 raw strings as filter options —
  unstable, noisy, and directly contradicting architecture §3's documented
  4-value enum.

## 3. Scope

### In scope

1. **Complete `STATUS_MAP` in `scrapers/verra/fetch.py`** covering all 17
   Verra `resourceStatus` values observed globally (11 of which appear in
   Indonesia today). Default is now `None` (NULL in DB) with a
   `status_unmapped` WARNING log — previously `"suspended"`. NULL is
   legitimate in the DB and visible to the T21 admin UI; silent
   misclassification was worse than explicit unknowns.

2. **Canonical value on `registries.status`.** One-line change in
   `upsert_registry_row()`: write `p.status_db` (already-mapped) instead
   of `p.status_raw`.

3. **Migration 003 — `scrapers/migrations/003_normalize_status.sql`:**
   defensive UPDATE that maps any lingering raw string on `projects.status`
   to the canonical enum (idempotent safety net), then adds a
   `CHECK (status IS NULL OR status IN ('active','pipeline','suspended','flagged'))`
   constraint. Future drift fails loudly at INSERT time rather than
   silently mislabelling data.

4. **Re-run Verra scraper** once to refresh all 64 projects' `status` +
   `registries.status` against the complete map.

5. **Re-run T09 scoring** to pick up the corrected `registries.status` for
   transparency sub-score computation.

### Out of scope (explicit non-goals)

- Normalizing `regulatory_events.importance` — separate CHECK constraint
  (via `DO $$ ASSERT $$` block in T10 seed); not touching it here.
- Revisiting T09 bucket thresholds — calibration work deferred to v0.2.
- Renormalizing `satellite_alerts.confidence` — already canonical
  (`high` / `nominal` / `low`).
- Extending the canonical enum itself — `{active,pipeline,suspended,flagged}`
  is architecture §3 and stable.

## 4. Acceptance criteria

**AC-1: every project row has a canonical or NULL status.**
```
SELECT DISTINCT status FROM projects
WHERE status IS NOT NULL
  AND status NOT IN ('active','pipeline','suspended','flagged');
```
Returns zero rows.

**AC-2: every registries row has canonical or NULL status.**
```
SELECT DISTINCT status FROM registries
WHERE status IS NOT NULL
  AND status NOT IN ('active','pipeline','suspended','flagged');
```
Returns zero rows.

**AC-3: CHECK constraint enforces canonical.**
```
SELECT conname FROM pg_constraint
WHERE conrelid='projects'::regclass AND contype='c';
```
Returns row `projects_status_canonical`.

**AC-4: T09 transparency differentiated.**
```
SELECT transparency_score, COUNT(*) FROM project_scores
WHERE score_date=CURRENT_DATE
GROUP BY transparency_score ORDER BY 1;
```
Returns ≥2 rows (previously 1 — uniform 55). Expected today:
`55 = 59, 70 = 5` (five projects with active Verra registry).

**AC-5: re-running the scraper is idempotent.**
```
python -m scrapers.verra.fetch
# records_updated: 64, errors: 0
```

**AC-6: known flagship projects correctly classified.**
```
SELECT slug, status FROM projects WHERE slug IN (
  'katingan-peatland-restoration-and-conservation-project',
  'rimba-raya-biodiversity-reserve-project',
  'sumatra-merang-peatland-project-smpp'
);
```
All three return `pipeline` (they are in Verra verification cycles, not
operationally "active" in the cycle-finished sense, and not "suspended").

**AC-7: migration 003 recorded.**
```
SELECT version FROM schema_migrations ORDER BY version;
# returns 001, 002, 003
```

## 5. Inputs & outputs

- **Inputs:** live Verra `resourceStatus` values for Indonesia, architecture §3 canonical enum.
- **Outputs:**
  - `scrapers/verra/fetch.py` — complete `STATUS_MAP`, `_map_status()` returns `None` on unknown, `upsert_registry_row()` writes `status_db`.

  - `docs/architecture.md` §3 — footnote referencing the CHECK constraint.

## 7. Follow-ups

None for v0.1. If a future Verra release introduces a new `resourceStatus`
