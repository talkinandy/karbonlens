### Fixed
- **T05 Google provider auto-discovery fix** ([fa51104]) тАФ `providers: [Google]` with no args relied on Auth.js v5 auto-detection from `AUTH_GOOGLE_ID` / `AUTH_GOOGLE_SECRET` env vars, but the runbook and `.env.example` use `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET`. Result: `client_id=undefined` in the OAuth authorization URL and `invalid_client` from Google on every sign-in attempt. Fixed by passing `clientId` / `clientSecret` explicitly to the `Google(...)` provider. T05 Phase B verified end-to-end on 2026-04-21: user + account + session rows populate correctly; `accounts.access_token` and `accounts.id_token` persist under snake_case JS keys (live validation of T04's adapter-field-name fix). Known residuals: `users.email_verified` not populated by the default adapter mapping; `accounts.refresh_token` empty because the Google provider isn't configured with `access_type=offline`. Neither blocks v0.1; tracked for v0.2.
- **T07 GFW query `Origin` header fix тАФ T07 Phase B complete** тАФ live run exposed that GFW validates the API key's domain allowlist against the request's `Origin` header, not just the key string. Without `Origin`, the query endpoint returned 403 "Request is missing valid API key" while the geostore POST succeeded (different auth policy). Added `Origin: http://localhost` to `register_geostore()` and `query_alerts()` headers (our key's allowlist is `localhost` + `karbonlens.netlify.app`). **Phase B live-verified 2026-04-21:** 246,576 satellite alerts ingested across 60 projects (4 of 64 had no alerts in the 90-day window), 60 in-app notifications fanned out to the signed-in user, 0 errors. T09 AC-5 now passes: Rimba Raya integrity_score = 62 тИИ [50, 65]; Katingan = 68 тИИ [60, 85]. Score distribution across all 64 projects: min 58, max 86, mean 65.4. T07 status flipped `done-phase-a` тЖТ `done`.
